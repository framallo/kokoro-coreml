# Kokoro Runtime Boundary for web-scraper TTS

**Date:** 2026-05-15
**Status:** Phase 0 boundary note

## Runtime Files

A native `web-scraper` Kokoro worker needs one pinned runtime bundle containing
these assets from `kokoro-coreml`:

- Core ML model packages under `coreml/`:
  - `kokoro_duration_t32.mlpackage`
  - `kokoro_duration_t64.mlpackage`
  - `kokoro_duration_t128.mlpackage`
  - `kokoro_duration_t256.mlpackage`
  - `kokoro_duration_t320.mlpackage`
  - `kokoro_duration_t384.mlpackage`
  - `kokoro_duration_t512.mlpackage`
  - optional exact-duration packages such as `kokoro_duration_exact_t44.mlpackage`
  - `kokoro_f0ntrain_t120.mlpackage`
  - `kokoro_f0ntrain_t280.mlpackage`
  - `kokoro_f0ntrain_t400.mlpackage`
  - `kokoro_f0ntrain_t600.mlpackage`
  - `kokoro_f0ntrain_t1200.mlpackage`
  - `kokoro_decoder_pre_3s.mlpackage`
  - `kokoro_decoder_pre_7s.mlpackage`
  - `kokoro_decoder_pre_10s.mlpackage`
  - `kokoro_decoder_pre_15s.mlpackage`
  - `kokoro_decoder_pre_30s.mlpackage`
  - `kokoro_decoder_har_post_3s.mlpackage`
  - `kokoro_decoder_har_post_7s.mlpackage`
  - `kokoro_decoder_har_post_10s.mlpackage`
  - `kokoro_decoder_har_post_15s.mlpackage`
  - `kokoro_decoder_har_post_30s.mlpackage`
- hn-NSF weights JSON generated by `scripts/prepare_swift_bench_inputs.py`
  or by the service proof command.
- Voice embeddings from `kokoro.js/voices/*.bin` or an equivalent pinned voice
  tensor artifact with stable per-voice hashes.
- Tokenizer and phonemizer assets from the same model bundle. The current
  Python reference uses `KPipeline`, `misaki`, and `espeak-ng`; the browser
  port uses `kokoro.js/src/phonemize.js` plus the `phonemizer` WASM package.
- A version manifest tying together the Swift package commit, model bundle,
  tokenizer/phonemizer assets, voice assets, hn-NSF weights, supported buckets,
  and MP3 encoder constraints.

## Swift Pipeline Contract

`KokoroPipeline.synthesize(...)` does not accept raw text. Its current public
input contract is:

```swift
public func synthesize(
    inputIds: [Int32],
    attentionMask: [Int32],
    refS: [Float],
    speed: Float = 1.0
) throws -> SynthesisResult
```

`inputIds` are Kokoro phoneme token IDs padded to a supported duration token
length, `attentionMask` marks real tokens versus padding, and `refS` is the
256-float voice embedding row selected for the phoneme string length. The
pipeline returns 24 kHz mono float audio plus timings and bucket metadata.

## v1 Responsibility Split

For `web-scraper` v1, raw text, voice, and speed enter the Cloudflare API. The
API validates request shape, stores raw text only in a private short-lived input
artifact, and gives the native worker a lease-bound input read route. The native
worker owns:

- text normalization and phonemization
- phoneme token ID mapping
- attention-mask construction
- voice embedding selection
- long-text chunking and bucket selection
- Core ML synthesis through `KokoroPipeline`
- WAV diagnostics
- MP3 encoding and encoder readiness checks

The Cloudflare API must not tokenize, compute embeddings, transcode audio, store
raw text in Durable Object state, or expose raw text in status/callback/logs.

## Phase 0 Proof Command

The local proof command is:

```bash
uv run python scripts/kokoro_service_proof.py --text "Hello world"
uv run python scripts/kokoro_service_proof.py --preset 30s
```

It generates one temporary pre-tokenized Swift input JSON, runs the existing
Swift/Core ML pipeline, writes a debug WAV, encodes MP3 with:

```bash
ffmpeg -hide_banner -nostdin -y -i input.wav -codec:a libmp3lame -b:a 128k -ar 24000 -ac 1 output.mp3
```

and writes a manifest containing text hash, voice, speed, token count, voice
embedding hash, Swift timings, encoder version, WAV hash, MP3 hash, MP3 probe
metadata, duration, and byte count.

This command intentionally uses the Python tokenizer as a Phase 0 proof bridge.
It is not the staging or production tokenizer boundary. The production worker
must ship tokenizer/phonemizer and voice embedding assets in the same signed,
pinned runtime bundle as the Core ML models.

## Parity Fixtures

The canonical bakeoff parity fixtures are:

- `outputs/swift_bench_inputs/3s.json`
- `outputs/swift_bench_inputs/7s.json`
- `outputs/swift_bench_inputs/15s.json`
- `outputs/swift_bench_inputs/30s.json`
- `outputs/swift_bench_inputs/hnsf_weights.json`

Each fixture records the original text, `af_heart` voice, speed, phonemes,
token IDs, attention mask, voice embedding, token count, and hn-NSF weights
hash. The service proof manifest adds MP3 metadata for the same boundary.

## Production Gap

The native worker still needs a production Swift-facing tokenizer and voice
embedding wrapper before staging. Until that exists, only the local proof path
is complete; the plan must not treat Python tokenization as production-ready.
